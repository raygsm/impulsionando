# Security Review — RLS Audit (2026-06-18)

## Summary

Rescanned the Supabase project after the consolidated RLS hardening migration.
All previously reported **ERROR** findings have been resolved and no new
**ERROR** finding was introduced on tables touched by the recent migrations.

| Severity | Before review | After review |
| --- | --- | --- |
| `error` (project-specific) | 1 (`companies` PII to anon) | **0** |
| `warn` (project-specific) | 1 (`comm_members` missing permission) | **0** |
| Framework baseline (extension in public, SECURITY DEFINER helpers, etc.) | 130 | 131 |

The remaining ~131 warnings are framework-wide noise from the linter:
`extension_in_public`, `rls_policy_always_true` on 3 internal infra policies,
and ~125 `Public/Auth Can Execute SECURITY DEFINER Function` items pointing
at shared helpers (`has_role`, `user_has_permission`, `user_belongs_to_company`,
etc.). These functions must remain callable — they are the building blocks of
every policy and dropping `EXECUTE` from `anon`/`authenticated` would break
RLS evaluation. Tracked as an accepted baseline in
[`security-memory.md`](../../mem/security-memory.md).

## Tables hardened in this review

| Table | Change |
| --- | --- |
| `companies` | Removed anon SELECT; replaced by `companies_vitrine_public` table populated by trigger with only non-PII columns |
| `comm_members` | Read now requires `comm.member.read` permission (self-read still allowed) |
| `quotes` | Read scoped to staff with `quotes.read`; INSERT cannot inject `public_token`, `status`, `accepted_*`, lead fields validated |
| `billing_pix_charges` | Read requires `billing.charge.read` |
| `contract_signatures` / `contract_documents` | Read requires `contracts.read` |
| `evt_events` | Read requires `evt.event.read` |
| `restaurant_table_invoices` | Read requires `restaurant.invoice.read` |
| `onboarding_checklist` | Read/write require `company.onboarding.read` / `company.onboarding.write` |
| `core_settings` | Read restricted to `is_impulsionando_staff` |
| `contab_onboarding`, `contab_irpf_journeys`, `contab_irpf_steps`, `contab_contracts` | Read requires the matching `contab.*` permission |
| `core_reschedule_rules`, `core_refund_rules` | Read requires `company.settings.read` |
| `billing_plans` | `internal_notes` no longer readable by `anon`/`authenticated` (column-level revoke) |
| `storage.objects` (`contracts`, `contab-documents` buckets) | SELECT now requires the same permission as the matching table |

## Recursion check

Searched every recently rewritten policy and confirmed that none reference
their own table directly. All cross-table checks are routed through the
existing `SECURITY DEFINER` helpers (`user_belongs_to_company`,
`user_has_permission`, `has_role`, `is_super_admin`, `is_impulsionando_staff`,
`core_user_belongs_to_company`). No recursive RLS detected.

## Grants audit

All newly created/changed policies were paired with the appropriate GRANT
posture:

- No new `GRANT … TO anon` was added to a table containing PII.
- The only anon-readable surface is `companies_vitrine_public` and the
  `vitrine_enabled = true` rows of `companies` projected through column-level
  grants (safe columns only — never `email`, `phone`, `document`, `owner_name`,
  `whatsapp`, etc.).
- `service_role` retains full access on every table.

## Automated guards added

- `tests/rls-recent-hardening.test.ts` — vitest suite that asserts anon
  cannot read any of the protected tables and cannot project PII columns from
  `companies`, while service-role can.
- `scripts/check-security-findings.mjs` — CI guard that runs a SQL audit and
  fails the build if either the count of `anon`-readable PII columns or the
  count of permissive `USING (true)` write policies exceeds the baseline in
  `scripts/security-baseline.json`. Run with `--update-baseline` to accept a
  deliberate change.
